# MapReduce и распределённая обработка: зачем Spark вообще появился

Главная идея: MapReduce показывает базовый принцип распределённой обработки. Большая задача разбивается на независимые куски, фаза `map` обрабатывает эти куски параллельно, затем shuffle группирует данные по ключу, а фаза `reduce` собирает итог. Spark не является Hadoop MapReduce, но многие идеи Spark проще понять, если сначала понять `map -> shuffle -> reduce`.

## Почему одной машине становится мало

Одна машина удобна, пока данных немного. Но в Data Engineering часто появляются задачи другого масштаба:

- логи за месяцы или годы;
- клики пользователей;
- заказы, платежи, события и транзакции;
- файлы в S3/HDFS/Data Lake;
- join и aggregation над десятками или сотнями гигабайт.

У одной машины есть ограничения: CPU, RAM, диск, сеть. Если файл на 2 ТБ не помещается на диск или не обрабатывается за разумное время, нужно распределить данные и вычисления.

## Что значит распределённая обработка

Распределённая обработка означает, что данные лежат частями на нескольких узлах, а вычисление выполняется параллельно рядом с этими частями.

Важная идея: дешевле отправить небольшой код туда, где уже лежат данные, чем перетаскивать большие данные на одну машину. Это называется data locality.

В Hadoop это особенно заметно: HDFS хранит блоки файлов на разных DataNode, а MapReduce старается запускать map tasks там, где лежат нужные блоки. Spark тоже учитывает locality, хотя работает на более общей модели execution plan и stages.

## Что такое MapReduce

MapReduce - это модель обработки данных в виде пар `key/value`.

Упрощённо:

- `map` берёт входные записи и превращает их в промежуточные пары `key/value`;
- shuffle/sort группирует все значения с одинаковым ключом;
- `reduce` получает ключ и список значений, затем считает итог.

Apache Hadoop Tutorial описывает это так:

```text
(input) <k1, v1> -> map -> <k2, v2> -> combine -> <k2, v2> -> reduce -> <k3, v3> (output)
```

Источник: Apache Hadoop MapReduce Tutorial: https://hadoop.apache.org/docs/stable/hadoop-mapreduce-client/hadoop-mapreduce-client-core/MapReduceTutorial.html

## Схема MapReduce на примере температур

![Using MapReduce to determine high temperatures per city](https://assets.ibm.com/is/image/ibm/using-mapreduce-to-determine-high-temperatures-per-city?ts=1766585529069&dpr=off)

Источник: IBM, What is MapReduce: https://www.ibm.com/think/topics/mapreduce

Что важно на этой схеме:

1. Input разбивается на части, чтобы несколько workers могли работать параллельно.
2. Mapping превращает входные записи в пары `key/value`.
3. Shuffling собирает одинаковые ключи вместе, например все значения для `Tokyo` или `London`.
4. Reducing агрегирует значения по ключу и выдаёт итог.
5. Result обычно записывается обратно в распределённое хранилище, а не собирается целиком в один процесс.

## Фаза Map

Map-фаза отвечает за локальную обработку входных данных. Каждый mapper читает свой кусок данных и независимо выпускает промежуточные пары.

Пример: хотим посчитать слова.

Входные строки:

```text
1: spark is fast
2: spark uses map reduce ideas
```

Map output:

```text
spark -> 1
is -> 1
fast -> 1
spark -> 1
uses -> 1
map -> 1
reduce -> 1
ideas -> 1
```

Map не считает глобальный итог. Mapper видит только свой кусок данных. Он может сделать локальную предварительную работу, но не знает, что происходит в других кусках.

## Фаза Shuffle/Sort

Shuffle - самая важная и самая дорогая часть. После map одинаковые ключи могут оказаться на разных узлах. Чтобы reducer мог посчитать итог по ключу, все значения этого ключа должны попасть в одно место.

После shuffle для word count данные становятся такими:

```text
spark -> [1, 1]
is -> [1]
fast -> [1]
uses -> [1]
map -> [1]
reduce -> [1]
ideas -> [1]
```

Почему shuffle дорогой:

- данные передаются по сети;
- промежуточные результаты сериализуются;
- данные сортируются и группируются по ключу;
- промежуточные файлы пишутся на диск;
- при нехватке памяти появляется spill;
- одна перекошенная группа может сделать один reducer намного медленнее остальных.

Это ключевой мост к Spark: когда в Spark вы видите `Exchange`, `Shuffle Read`, `Shuffle Write`, `SortMergeJoin` или stage boundary, это та же базовая проблема перераспределения данных по ключу.

## Фаза Reduce

Reduce получает ключ и список значений. Он считает итог.

Для word count:

```text
spark -> [1, 1] -> spark -> 2
is -> [1] -> is -> 1
fast -> [1] -> fast -> 1
```

Reducer обычно не видит весь dataset целиком. Он видит свою часть key space. Например, один reducer может отвечать за ключи `A-M`, другой - за `N-Z`, или partitioning может быть hash-based.

## Combiner: локальная агрегация до shuffle

В классическом MapReduce можно использовать combiner. Combiner выполняет локальную агрегацию на стороне mapper, чтобы уменьшить объём shuffle.

Например, mapper прочитал строку:

```text
spark spark spark
```

Без combiner он отправит:

```text
spark -> 1
spark -> 1
spark -> 1
```

С combiner он может отправить:

```text
spark -> 3
```

Это не отменяет shuffle, но уменьшает количество данных, которые надо передать по сети.

В Spark похожая идея есть в partial aggregation: перед shuffle Spark часто делает частичную агрегацию внутри partitions, а затем финальную агрегацию после shuffle.

## Что MapReduce делает хорошо

MapReduce хорошо подходит для задач, где данные можно обработать большими независимыми проходами:

- word count;
- подсчёт метрик по ключу;
- обработка логов;
- простые ETL-задачи;
- агрегация больших файлов;
- подготовка данных к дальнейшему анализу.

Сильные стороны:

- масштабируется на много машин;
- устойчив к сбоям tasks;
- работает с большими файлами;
- хорошо ложится на batch processing.

## Ограничения классического MapReduce

MapReduce неудобен, когда вычисление состоит из многих шагов.

Например:

```text
read -> filter -> join -> groupBy -> join -> sort -> write
```

В классическом Hadoop MapReduce это часто превращается в цепочку отдельных jobs. Каждый job читает данные из HDFS и пишет промежуточный результат обратно в HDFS. Это надёжно, но дорого.

Проблемы:

- много промежуточной записи на диск;
- сложно писать и поддерживать код;
- тяжело делать интерактивную аналитику;
- итеративные алгоритмы, например ML, работают медленно;
- оптимизатор не видит всю цепочку как единый план.

## Как Spark развивает эту идею

Spark сохраняет базовую идею распределённой обработки, но делает модель гибче.

В Spark:

- есть transformations и actions;
- строится DAG вычислений;
- Spark может оптимизировать план до запуска;
- данные можно cache/persist в памяти;
- одна программа может содержать длинную цепочку операций;
- DataFrame API и Catalyst optimizer прячут часть низкоуровневой сложности;
- shuffle всё равно остаётся дорогой операцией.

Spark не отменяет MapReduce-мышление полностью. Он делает его более выразительным. Когда вы пишете:

```python
orders.groupBy("customer_id").count()
```

Spark всё равно должен сделать идею `map -> shuffle -> reduce`: сначала обработать partitions, затем собрать одинаковые `customer_id`, затем посчитать итог.

## Что запомнить перед изучением Spark

- Распределённая обработка нужна, когда одна машина не справляется с объёмом данных или временем выполнения.
- Данные режутся на части, чтобы обрабатывать их параллельно.
- `map` обрабатывает свою часть данных независимо.
- shuffle группирует одинаковые ключи вместе и поэтому дорогой.
- `reduce` считает итог по ключу.
- Spark развивает эту идею через DataFrame API, DAG, stages, tasks и Catalyst optimizer.
- Если вы понимаете, почему MapReduce делает shuffle, вам проще понять, почему Spark делает `Exchange`.
